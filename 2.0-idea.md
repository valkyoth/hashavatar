• The 1.1.2 codebase is a solid security-conscious foundation, but the rendering core is not yet a suitable base for a bit-identical 2.0 contract. The decisive change
  should be to make one fixed-point, validated scene the source of truth for SVG, CPU raster, and any GPU backend.

  During this audit:

  - cargo test --locked: 95 tests passed, 1 ignored, plus the doctest.
  - AArch64 and wasm32-unknown-unknown cross-compilation succeeded.
  - All 5 Kani harnesses passed. The documentation currently says 4, so that count is stale.
  - The worktree remains unchanged.

  ## 1. Architectural Pipeline & Rendering Engine Hardening (v2.0 Core)

  ### Current architectural assessment

  AvatarRenderPlan is already the right conceptual seam: it centralizes validated size, identity, and style selection. However, it dispatches separately into family-
  specific raster implementations and family-specific SVG implementations (src/api.rs:138, src/api.rs:448).

  Those implementations are not two serializations of the same geometry. For example:

  - Raster Cat derives a CatGenome, seeded RNG, family palette, and several float-derived measurements (src/avatars/cat.rs:36).
  - SVG Cat independently chooses dimensions and colors directly from other digest bytes (src/svg/cat.rs:3).
  - Decorative raster backgrounds adapt pattern sizes to the canvas, while SVG backgrounds use fixed literal pattern dimensions (src/api.rs:476).

  Consequently, 1.x SVG is an alternate visual rendition, not a lossless vector representation of the raster artwork.

  ### Proposed 2.0 pipeline

  AvatarRequest
      │
      ├── Identity derivation / pseudonymization
      ├── Versioned deterministic trait derivation
      └── Family + style selection
               │
               v
         Hierarchical FamilyRig
               │
         Accessory resolver
               │
         Scene graph compiler
               │
         Validated fixed-point scene
            ┌──┴───────────────┬─────────────────────┐
            v                  v                     v
       SVG writer      Reference CPU rasterizer   GPU accelerator
                              │
                         RGBA surface
                              │
                      optional format encoder

  The scene should be the only artistic source of truth. Family modules should compile semantic primitives, never draw pixels or concatenate XML.

  A practical internal representation is:

  - SceneGraph: authoring-level groups, transforms, clips, opacity groups, semantic layer IDs, and attachment metadata.
  - DisplayList: validated, flattened commands in final compositing order.
  - Separate fixed-capacity arenas for commands, path verbs, points, paints, transforms, and collision hulls.
  - Commands such as FillPath, StrokePath, FillEllipse, FillRect, PushClip, PopClip, and BeginOpacityGroup.
  - No external images, arbitrary SVG fragments, shaders, filters, or unbounded recursion.

  Every scene must be validated before backend execution: arena ranges, stack balance, coordinate bounds, command count, path complexity, clip depth, and estimated raster
  cost.

  ### Zero-allocation qualification

  A dynamically extensible retained scene cannot be unconditionally zero-allocation. The safe design is storage-parametric:

  - BorrowedScene<'a> writes into caller-provided, preinitialized command and point slices.
  - OwnedScene uses Vec under alloc.
  - SceneSink streams commands directly when retention is unnecessary.
  - Every family publishes a conservative SceneBudget.
  - Insufficient caller storage returns SceneCapacityExceeded, never grows silently.

  This keeps #![forbid(unsafe_code)] and permits bare-metal callers to statically provision storage. Using MaybeUninit internally would require an unsafe boundary;
  preinitialized command slices or an audited fixed-capacity dependency avoid that.

  ### Fixed-point geometry and canonical rasterization

  Float use is pervasive today: anchors, HSL conversion, family geometry, Wu lines, ellipses, polygon intersections, and trigonometric curve sampling. Representative
  examples are the float Bresenham/Wu implementation and f64 ellipse algorithm in src/primitives.rs:129, polygon intersection division in src/primitives.rs:450, and
  trigonometric layer curves in src/layers.rs:718.

  Recommended numeric contract:

  - Coord(i32): signed Q16.16 for scene coordinates.
  - Unit(u16): Q0.16 for normalized traits, interpolation, and anchor coordinates.
  - Coverage(u16): canonical coverage accumulator, rounded once to RGBA8.
  - i64 intermediates for coordinate multiplication and transforms.
  - i128 only for proofs or operations whose worst-case bounds exceed i64.
  - One documented signed division and rounding rule, using Euclidean quotient/remainder rather than casts.
  - Checked construction at trust boundaries; do not use silent saturation to conceal invalid scene geometry.

  Q16.16 provides much more subpixel precision than needed at the existing 2048-pixel maximum while retaining adequate signed range. Q24.8 would simplify 8-bit coverage
  but is less attractive for repeated transform composition. A useful compromise is Q16.16 scene geometry with an 8×8 or 16×16 integer sample grid in the reference
  rasterizer.

  Curves should use deterministic, bounded de Casteljau subdivision or fixed-point forward differencing. Subdivision must have:

  - An integer flatness metric.
  - A fixed maximum depth.
  - An explicit stack rather than recursive call-stack dependence.
  - A defined tie-breaking rule.
  - Stroke lowering into deterministic fill geometry when necessary.

  Replace palette-based float HSL conversion with a small integer sRGB palette module. Replace StdRng with a frozen algorithm or, preferably, stateless domain-separated
  trait derivation such as derive_u16("cat/head-width", counter). StdRng is also the current zeroization residual and should not form part of a permanent visual contract.

  ### What “bit-identical” should mean

  Define three separate contracts:

  1. SceneDigest: canonical serialization of the validated scene is identical.
  2. PixelDigest: canonical CPU rasterization produces identical raw premultiplied RGBA8 bytes.
  3. EncodedAssetDigest: only promised for explicitly canonical encoders and encoder versions.

  Lossless WebP or PNG can decode to identical pixels without producing identical encoded bytes after dependency upgrades. AVIF/JXL especially should not enter the
  encoded-byte stability contract.

  SVG emission can be lossless relative to scene semantics by formatting fixed-point values without floating point and preserving stable element/attribute order. Browser-
  rendered SVG pixels cannot be promised identical to the crate’s canonical rasterizer.

  For GPU:

  - Do not initially make hardware blending, floating shaders, or vendor multisampling canonical.
  - Either classify GPU output as “visually conforming” or use integer compute shaders and storage textures with software-defined compositing.
  - Promote a GPU backend to bit-identical status only after it passes the full raw-pixel corpus across multiple vendors and drivers.
  - Retain the safe-Rust CPU rasterizer as the normative backend.

  Use premultiplied color internally and define exact integer Porter–Duff source-over math. The current blend path is deterministic integer arithmetic but is not an
  adequate general compositing specification for translucent destinations (src/layers.rs:315).

  ### Hierarchical anchors and multi-accessory stacking

  The present anchor table is a flat face model containing two eyes, mouth, top, neck, face width, and eye radius (src/layers.rs:22). Eleven families return no anchors
  and silently skip accessories and expressions (src/layers.rs:232). AvatarStyleOptions contains exactly one accessory (src/model.rs:805).

  Replace this with a hierarchical FamilyRig:

  - Canvas root → body → neck → head.
  - Head → crown, forehead, eye bridge, individual eyes, nose, mouth, ears.
  - Body → shoulders, hands, handheld origins, lower-body anchors.
  - Each node has a fixed-point local transform inherited from its parent.
  - Families expose a capability mask and calibrated safe zones.
  - Non-face families simply lack FaceRig/EyeRig, rather than returning a generic None.

  Typed slots should include at least:

  - BackAccessory
  - HeadAura
  - Headwear
  - Earwear
  - Facewear
  - Eyewear
  - Neckwear
  - HandheldLeft
  - HandheldRight
  - ForegroundEffect

  Each accessory declares its required anchor, local collision hulls, exclusion zones, allowed scaling range, deterministic displacement candidates, and semantic z-band.

  Resolution should be deterministic:

  1. Canonicalize by slot and stable accessory ID, not caller insertion order.
  2. Validate family capabilities.
  3. Place higher-priority structural slots first.
  4. Try a fixed sequence of offsets/scales.
  5. Reject placements outside the frame-safe region or intersecting exclusion hulls.
  6. Return a LayoutReport with accepted, adjusted, rejected, or skipped layers.
  7. Sort by (ZBand, slot ordinal, stable accessory ID).

  Manual styles should default to an error for unsupported combinations. Automatic styles may use a documented deterministic fallback policy. Silent skipping, which
  currently happens at src/layers.rs:371, is too surprising for a production manual API.

  ## 2. Developer Experience (DX), Ergonomics, & API Modernization

  ### Typestate builder

  The current builder stores a runtime AvatarKind, accepts any accessory, and validates compatibility only implicitly during rendering (src/api.rs:520).

  Rust typestate cannot infer a compile-time family type from .kind(AvatarKind) because the argument is a runtime enum. Offer two complementary APIs:

  AvatarBuilder::for_identity(identity)
      .cat()
      .headwear(Headwear::Crown)
      .eyewear(Eyewear::RoundGlasses)
      .render_into(&mut surface)?;

  AvatarBuilder::for_identity(identity)
      .kind(dynamic_kind)
      .accessories(dynamic_stack)
      .validate()?
      .render_into(&mut surface)?;

  The static path uses marker types such as CatFamily, PlanetFamily, and sealed capability traits like SupportsFacewear and SupportsHandheld. A PlanetFamily builder
  simply has no .eyewear() method. The dynamic path remains appropriate for JSON, database, and URL-derived settings and returns a detailed StyleCompatibilityError.

  Avoid full typestate for size, background, identity, every slot, and output format—it would make compiler errors and type signatures unmanageable. Family capability is
  the typestate boundary that provides the most value.

  Also address two existing extensibility problems:

  - Public enums are not actually #[non_exhaustive], although the stability policy assumes callers can tolerate new variants (docs/STABILITY.md:6).
  - Automatic selection uses modulo over ALL.len(), so adding a variant reshuffles existing identities (src/model.rs:847).

  In 2.0, use #[non_exhaustive] for built-ins and stable string/newtype IDs for external extensions. Automatic selection must use an explicit CatalogVersion with frozen
  discriminants and weights. Adding art creates a new catalog version; it must not mutate the old mapping.

  ### Zero-copy and streaming APIs

  The current public encode path necessarily returns Vec<u8>, and the public renderer returns RgbaImage. A writer-based encoder already exists internally but is not
  exposed (src/encoding.rs:87).

  Recommended APIs:

  let mut surface = RgbaSurfaceMut::new(&mut pixels, width, height, stride)?;
  avatar.render_into(&mut surface)?;

  let written = avatar.encode_into_slice(
      AvatarOutputFormat::Png,
      &mut encoded,
  )?;

  avatar.encode_to_writer(
      AvatarOutputFormat::WebP,
      &mut writer,
  )?;

  avatar.write_svg(&mut fmt_writer)?;

  Important distinctions:

  - A caller-provided &mut [u8] is sufficient for synchronous rendering; Pin provides no benefit unless the API retains pointers for async DMA/GPU work.
  - Raster rendering still requires frame memory unless the rasterizer supports scene-based scanline or tile replay.
  - Writing encoded bytes directly to std::io::Write avoids the output Vec, but codecs may still allocate internally.
  - core::fmt::Write is suitable for SVG. A small crate-native ByteSink trait can support binary output in no_std.
  - Slice encoders should return OutputBufferTooSmall { required: Option<usize>, written }.
  - Encoder capability metadata should expose requires_full_frame, maximum scratch estimate, streaming support, alpha support, and deterministic-byte status.

  The current polygon rasterizer allocates a closed polygon, an edge vector, and an intersections vector per call (src/primitives.rs:463). The fixed-point tile rasterizer
  should instead reuse caller-provided scratch arrays or a render arena.

  ### no_std + alloc

  Despite default = [], the current crate is not a minimal/no-std build: image, palette/std, rand/std, and WebP are mandatory dependencies (Cargo.toml:19), while the
  crate directly imports std, RgbaImage, codecs, and StdRng (src/lib.rs:31).

  Recommended feature topology:

  default       = std + image-compat + webp
  alloc         = owned scenes, String conveniences
  std           = alloc + std::io adapters + errors
  raw-rgba      = core raster surface API
  svg           = SVG sink; String helper requires alloc
  image-compat  = RgbaImage adapters
  webp/png/...  = std codec adapters
  serde         = no_std-compatible enum/request model
  json-schema   = alloc adapter
  web-query     = alloc or caller-scratch parser
  gpu           = std-only, noncanonical initially

  Required migration work:

  - Add #![cfg_attr(not(feature = "std"), no_std)] and extern crate alloc.
  - Replace std::mem, std::fmt, and std::error usage with core plus gated std::error::Error.
  - Move all image types and encoders behind image-compat/codec features.
  - Own the raw pixel surface abstraction.
  - Replace palette with integer color conversion.
  - Replace StdRng with the frozen no-std trait derivation system.
  - Stream length-prefixed identity fields directly into hash state rather than allocating a concatenated SecretVec preimage (src/core.rs:527).
  - Offer borrowed/fixed-capacity scenes for heapless operation and OwnedScene under alloc.
  - Run CI for --no-default-features, no_std + alloc, WASM, and at least one embedded target.

  Aesynx or another microkernel only needs an allocator for the alloc profile and a caller-provided output sink. The core must not depend on filesystem, threads, entropy,
  clocks, or OS I/O.

  ### JSON Schema and web query adapters

  Keep these adapters separate from the renderer:

  - AvatarRequest as the strict validated domain model.
  - JSON Schema behind an optional schema feature.
  - Query parser with bounded key count, bounded decoded length, duplicate-key rejection, unknown-key policy, and strict enum errors.
  - TryFrom<AvatarQuery> returns a dynamic validated builder/request.
  - No silent fallback for malformed values.
  - Generated schemas must include catalog/render-contract identifiers and slot compatibility metadata.

  ## 3. Security, Privacy, & High-Assurance Hardening

  ### Native keyed pseudonymization

  Current identity construction is length-bounded, length-prefixed, domain-separated, constant-time comparable, redacted in Debug, and sanitized on drop. These are strong
  controls. However, cache_key() correctly documents that unkeyed rehashing does not stop offline enumeration (src/core.rs:429).

  Add a first-class keyed constructor:

  let identity = AvatarIdentity::from_keyed(
      &identity_key,
      IdentityDomain::new("tenant-a", "profile-avatar", "pseudonym-v1")?,
      email.as_bytes(),
  )?;

  Recommended construction:

  - HKDF-SHA-256 derives a purpose/tenant subkey from caller-managed master material.
  - HMAC-SHA-256 derives the pseudonym from the bounded normalized identifier.
  - Alternatively, keyed BLAKE3 may be an explicitly identified algorithm.
  - Every preimage includes a fixed protocol label, algorithm ID, version, tenant, purpose, and length-prefixed input.
  - IdentityKey has redacted Debug, no serialization, no accidental Clone, constant-time handling where relevant, and zeroization on drop.
  - Key IDs are public rotation metadata; key bytes never enter cache keys or scenes.
  - Raw email normalization remains an application policy, because case folding and provider-specific equivalence cannot safely be guessed by the crate.

  Separate:

  - IdentityCacheKey: identifies the pseudonymous subject.
  - AvatarAssetKey: additionally binds render contract, catalog version, dimensions, seed, style stack, backend pixel contract, and output format.

  The existing cache key only identifies AvatarIdentity, not a complete rendered asset.

  Security language should say “pseudonymization,” not “anonymization.” Keying prevents practical dictionary testing while the key remains secret, but:

  - Equal inputs remain linkable within a domain.
  - Key compromise restores dictionary testing.
  - The avatar itself is a stable correlator.
  - Small populations and visible traits can leak membership information.
  - No construction can promise regulatory anonymity by itself.

  ### Zeroization

  Current strengths:

  - AvatarIdentity scrubs its digest.
  - Hash preimages use SecretVec.
  - Encoded output is guarded until successful return.
  - Owned completed rasters are wrapped and wiped after encoding (src/encoding.rs:18).
  - JPEG flattening uses a guarded temporary buffer.

  Remaining gaps:

  - AvatarBuilder<T> retains the original identity value until rendering and cannot generically wipe borrowed or caller-owned T (src/api.rs:520).
  - The raster receives its sanitizing wrapper only after family rendering completes. A panic during drawing drops an ordinary RgbaImage.
  - Pixel scrubbing covers the logical slice; allocator capacity and codec-owned scratch require separate guarantees.
  - StdRng expanded state is not wipeable.
  - Third-party WebP/PNG/GIF internals may allocate inaccessible scratch memory.
  - Returned buffers necessarily remain caller-owned.

  For 2.0:

  - Construct SensitiveCanvas at allocation time, before any pixels are written.
  - Use SensitiveScene, SensitiveScratch, and SensitiveOutput arena guards.
  - Eagerly derive keyed identity so high-assurance builders store only the pseudonym/digest.
  - Remove stateful StdRng.
  - Provide an optional SecretOutput<Vec<u8>> convenience and clear ownership documentation for normal outputs.
  - For caller-provided memory, offer a guard type that wipes on drop; a plain borrowed slice cannot be wiped automatically after returning control without destroying the
    caller’s output.

  - For GPU buffers, submit an explicit zero-fill and wait for the completion fence before pooling or releasing sensitive resources.
  - If codec scratch must be covered, use a codec with allocator hooks into a wiping arena, an audited fork, or process/linear-memory isolation followed by whole-arena
    wiping.

  The guarantee should be phrased as: “all hashavatar-owned sensitive allocations are scrubbed on normal return, error, and unwinding.” Process aborts, copies in
  registers, external codec allocations, OS paging, and hardware copies remain out of scope unless separately controlled.

  ### Expanded Kani coverage

  The current proofs cover dimension/resource arithmetic and rectangle invariants (src/kani_proofs.rs:3). All 5 passed during this audit, although docs/KANI.md:8 says 4.

  Add proof groups for:

  - Fixed-point construction, add/subtract, multiplication, division, rounding, and transform composition.
  - Integer palette interpolation and alpha compositing: output channels always remain in range.
  - Curve subdivision termination, maximum stack depth, and endpoint preservation.
  - Edge setup and tile bounds.
  - Pixel index calculations for arbitrary valid width, height, stride, tile, and channel.
  - Path arena ranges and scene validation.
  - Clip/transform stack balance.
  - Frame masks for every valid dimension and boundary coordinate.
  - Polygon crossing arithmetic and fill-rule behavior.
  - Accessory resolver termination, deterministic sorting, and capacity bounds.

  Keep proofs small and pure. Model a tile or scanline, not a complete 2048² render. Use proof contracts for validated inputs and explicit unwind bounds for curve
  subdivision.

  At rc.1, Kani must fail closed. The current script treats missing/incompatible Kani as success (scripts/check_kani.sh:4); that is reasonable for normal development but
  insufficient as an RC assurance gate.

  Kani should remain complemented by:

  - Differential fuzzing between reference and optimized CPU backends.
  - Scene parser/validator fuzzing.
  - Accessory-stack and collision fuzzing.
  - Miri for unsafe-free core assumptions.
  - Sanitizer builds for dependency adapters.
  - Cross-architecture raw-RGBA golden tests.
  - Codec decode-and-compare tests.
  - SVG parsing and scene-to-SVG structural tests.

  The current CI runs one Ubuntu x86_64 job (.github/workflows/ci.yml:20). The RC matrix needs native AArch64 execution, WASM execution, multiple optimization/LTO
  profiles, and ideally a big-endian supplemental target—not merely cross-compilation.
